# 🧹 FinanZen — Script de Auditoria Clean Code

> **Propósito:** Este documento é um checklist vivo de auditoria técnica.
> Ele deve ser executado sempre que um ciclo de desenvolvimento se encerrar,
> ou antes de qualquer merge significativo, para garantir aderência às regras
> de Clean Code, Segurança e Observabilidade definidas no `user_global`.
>
> **Como usar:** Copie este arquivo, marque os itens como `[x]` conforme
> for resolvendo, e ao final registre a data de conclusão na seção de Histórico.

---

## 🔴 Severidade CRÍTICA (Bloqueiam deploy)

### Segurança & Compliance
- [x] **Zero Secrets em Logs:** Verificar que nenhum `logger.info/warn/error` expõe API Keys, senhas de banco, tokens ou PIIs (nomes, e-mails, telefones de pagantes). Referência LGPD.
- [x] **Sanitização de Inputs:** Toda rota `POST/PATCH` que recebe `body` do cliente valida via schema declarativo (`zod`) antes de tocar no banco. Inputs crus nunca devem ir direto ao Prisma.
- [x] **Catch Blocks Vazios:** Nenhum `catch (e) {}` silencioso deve existir. Toda exceção deve ser registrada com nível `ERROR` ou `WARN`.
- [ ] **CORS Restritivo:** `origin: "*"` é aceitável apenas em `development`. Em `production`, o CORS deve listar domínios explícitos.

### Tratamento de Erros (Fail Fast)
- [x] **Error Handler Global:** O plugin `errorHandlerPlugin` está registrado e cobre `ZodError` + erros genéricos, retornando JSON padronizado com `statusCode`, `error` e `message`.
- [x] **Try/Catch em Todas as Rotas:** Cada handler de rota possui `try/catch` com log estruturado (`traceId`, `error.message`). Nenhuma rota lança exceção não capturada.

## 🟡 Severidade ALTA (Dívida técnica ativa)

### Arquitetura & SRP (Responsabilidade Única)
- [x] **Arquivo Monolítico:** `server.ts` contém todas as rotas (webhook, transactions, categories, limits, message-logs). Extrair em módulos: `routes/webhook.ts`, `routes/transactions.ts`, `routes/categories.ts`, `routes/limits.ts`.
- [x] **Componente Monolítico:** `DashboardClient.tsx` possui ~870 linhas e 5 abas inteiras. Extrair cada aba em componente próprio: `ChatTab.tsx`, `CategoriesTab.tsx`, `AlertsTab.tsx`, `SettingsTab.tsx`.
- [x] **URL Hardcoded:** A string `http://127.0.0.1:3001` aparece repetida em múltiplos `fetch()` no frontend. Centralizar em uma constante `API_BASE_URL` ou variável de ambiente `NEXT_PUBLIC_API_URL`.

### Tipagem Obrigatória (Type System)
- [x] **Eliminar `any`:** O tipo `any` é usado em props do componente (`fluxoCaixa: any[]`, `categorias: any[]`, `transacoes: any[]`), em `useState<any>`, e em `request.body as any`. Criar interfaces tipadas: `Transaction`, `Category`, `MessageLog`, `BudgetLimit`.
- [x] **Retornos de Rota Tipados:** As rotas do backend não declaram schemas de `Body`, `Params` nem `Response`. Usar o sistema de schemas do Fastify com Zod para validação automática.

### Observabilidade (Logs)
- [x] **`console.error` Residual:** O frontend usa `console.error(err)` em `useEffect` de fetch. Substituir por tratamento adequado (notificação ao usuário) ou remover.
- [x] **Trace IDs Consistentes:** Verificar que TODAS as rotas incluem `traceId: request.id` nos logs de `ERROR` e `INFO`. As rotas de categories/limits/message-logs não incluem `traceId`.

## 🟢 Severidade MÉDIA (Refino de qualidade)

### Clean Code & DRY
- [x] **Imports Não Utilizados:** `LineChart`, `Line`, `CartesianGrid` são importados de `recharts` mas nunca usados. O `z` de `zod` é importado no `server.ts` mas nunca utilizado. Remover.
- [x] **Lógica de Fetch Duplicada:** Os `useEffect` de `fetchLogs` e `fetchLimits` seguem padrão idêntico (set loading → fetch → set data → catch → finally). Extrair um hook genérico `useFetchOnTab(url, tab)`.
- [x] **`window.location.reload()` Excessivo:** Após cada operação CRUD, o frontend recarrega a página inteira. Substituir por invalidação de estado local (re-fetch seletivo) para UX profissional.
- [x] **Magic Strings:** Valores como `"MONTHLY"`, `"PROCESSED"`, `"FAILED"`, `"INCOME"`, `"EXPENSE"` estão espalhados como strings literais. Centralizar em `enums` ou `const objects`.

### SEO & Metadados
- [x] **Metadata Padrão:** O `layout.tsx` ainda possui `title: "Create Next App"` e `description: "Generated by create next app"`. Atualizar para refletir o FinanZen.
- [x] **Lang Incorreto:** O atributo `lang="en"` deve ser `lang="pt-BR"`.

### UI/UX
- [x] **Feedback de Erro nos Fetches:** Quando um `fetch` falha nos `useEffect`, o erro é silenciosamente logado no console. O usuário não recebe nenhum feedback visual.
- [x] **Dados Mockados Residuais:** O número de WhatsApp `+55 61 99833-2518` está hardcoded na aba de Configurações. Deve ser buscado do banco.

## ⚪ Severidade BAIXA (Polish & Evolução)

### Testes
- [ ] **Cobertura Mínima (80%):** Não existem testes unitários para `extractFinancialData`, `parseWebhookPayload` ou `sendWhatsAppMessage`. Criar suíte com mocks para validar parsing de JSON da IA.
- [ ] **Testes de Integração:** Criar cenários de integração para as rotas CRUD (transactions, categories, limits) usando banco em memória ou fixtures.

### Performance
- [ ] **Re-renders Desnecessários:** O componente `DashboardClient` re-renderiza inteiro a cada mudança de estado. Avaliar `useMemo` para cálculos derivados e `React.memo` para sub-componentes.
- [ ] **Instância Intl:** `formatCurrency` cria uma nova instância de `Intl.NumberFormat` a cada chamada. Mover para fora do componente como constante.

### Documentação
- [ ] **Comentários de Motivação:** Garantir que comentários explicam o "porquê" de decisões de negócio, não o "como" do código. Remover comentários óbvios tipo `// Format currency`.
- [ ] **README Atualizado:** Refletir as novas funcionalidades (Chat, Categorias, Alertas, Configurações) no `README.md`.
